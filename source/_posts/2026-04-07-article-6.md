## 引言

Nginx 是反向代理和负载均衡的事实标准。但很多人只用了它的皮毛——静态文件托管或简单代理。

真正的 Nginx 能力远不止这些。它能帮你：
- 将流量均匀分发到多个后端服务
- 在 HTTPS 加密下无缝切换证书
- 缓存静态内容，减轻后端压力
- 实时监控服务健康状态

这篇指南，我会用**实际配置案例**带你从基础到生产级，搞清楚 Nginx 反向代理与负载均衡的每一个关键配置。

---

## 一、反向代理基础

### 1.1 为什么需要反向代理？

```
客户端 → Nginx（反向代理） → 后端服务
```

客户端只和 Nginx 通信，后端服务藏在 Nginx 后面，好处：
- **安全**：不暴露后端 IP 和端口
- **统一入口**：一个域名对应多个服务
- **SSL 终止**：Nginx 处理 HTTPS，后端只需 HTTP
- **缓存**：静态资源直接返回，不经过后端

### 1.2 最简单的代理配置

```nginx
server {
    listen 80;
    server_name api.example.com;

    location / {
        proxy_pass http://backend_server:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
```

关键指令：
- `proxy_pass`：目标后端地址
- `proxy_set_header`：传递客户端信息（Host、IP）

## 二、负载均衡配置

当单个后端不够时，你需要多个实例分担流量。

### 2.1 upstream 块定义后端池

```nginx
upstream backend {
    server 10.0.1.10:8080 max_fails=3 fail_timeout=30s;
    server 10.0.1.11:8080 max_fails=3 fail_timeout=30s;
    server 10.0.1.12:8080 backup;  # 备用节点
}

server {
    listen 80;
    server_name api.example.com;

    location / {
        proxy_pass http://backend;
    }
}
```

### 2.2 负载均衡算法

默认是 **轮询（round-robin）**，每个请求依次分配到不同后端。

其他算法：
- **weight**：按权重分配（性能好的服务器权重高）
- **ip_hash**：同一客户端 IP 固定到一台后端（会话保持）
- **least_conn**：最少连接数优先
- **hash**：自定义 key 哈希（如 `$request_uri`）

**示例：权重分配**

```nginx
upstream backend {
    server 10.0.1.10:8080 weight=3;  # 3/6 的请求
    server 10.0.1.11:8080 weight=3;  # 3/6 的请求
}
```

## 三、健康检查与故障转移

### 3.1 被动健康检查

使用 `max_fails` 和 `fail_timeout`：

```nginx
upstream backend {
    server 10.0.1.10:8080 max_fails=3 fail_timeout=30s;
}
```

含义：
- 在 `fail_timeout` 时间内，如果请求失败达到 `max_fails` 次，该后端被标记为宕机
- 宕机时间过后，会重新尝试恢复

### 3.2 主动健康检查（Nginx Plus 或 OpenResty）

社区版 Nginx 不支持主动探活，但可以用 `ngx_http_upstream_check_module` 模块（需要重新编译）。

生产环境常见做法：**使用 Nginx Unit 或 HAProxy 做健康检查，Nginx 仅做代理**。

## 四、SSL 加密配置

### 4.1 Let's Encrypt 免费证书

使用 Certbot 自动获取：

```bash
certbot --nginx -d api.example.com
```

Certbot 会自动修改 Nginx 配置，添加 `ssl_certificate` 路径。

### 4.2 手动 SSL 配置

```nginx
server {
    listen 443 ssl http2;
    server_name api.example.com;

    ssl_certificate /etc/ssl/certs/example.com.crt;
    ssl_certificate_key /etc/ssl/private/example.com.key;

    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers ECDHE-RSA-AES256-GCM-SHA512:DHE-RSA-AES256-GCM-SHA512;

    # OCSP 装订，加速握手
    ssl_stapling on;
    ssl_stapling_verify on;

    location / {
        proxy_pass http://backend;
        proxy_set_header X-Forwarded-Proto https;
    }
}
```

## 五、缓存与性能优化

### 5.1 静态资源缓存

后端返回的静态文件（图片、CSS、JS）可以在 Nginx 缓存，减少后端压力。

```nginx
proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=my_cache:10m inactive=60m;

server {
    ...

    location /static/ {
        proxy_cache my_cache;
        proxy_cache_valid 200 302 10m;   # 缓存 10 分钟
        proxy_cache_valid 404 1m;
        add_header X-Cache-Status $upstream_cache_status;
        proxy_pass http://backend;
    }
}
```

`$upstream_cache_status` 会返回 `HIT`（命中）或 `MISS`（未命中）。

### 5.2 压缩传输

启用 gzip 压缩：

```nginx
gzip on;
gzip_vary on;
gzip_min_length 1024;
gzip_types text/plain text/css application/json application/javascript;
```

## 六、完整生产配置示例

一个典型的 API 网关配置：

```nginx
# 上游后端池
upstream api_backend {
    least_conn;  # 最少连接优先
    server 10.0.1.10:8080 max_fails=3 fail_timeout=30s;
    server 10.0.1.11:8080 max_fails=3 fail_timeout=30s;
    server 10.0.1.12:8080 backup;
}

# HTTP → HTTPS 重定向
server {
    listen 80;
    server_name api.example.com;
    return 301 https://$server_name$request_uri;
}

# HTTPS 主服务
server {
    listen 443 ssl http2;
    server_name api.example.com;

    # SSL 配置
    ssl_certificate /etc/letsencrypt/live/api.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/api.example.com/privkey.pem;
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers ECDHE-RSA-AES256-GCM-SHA512:DHE-RSA-AES256-GCM-SHA512;
    ssl_stapling on;
    ssl_stapling_verify on;

    # 访问日志
    access_log /var/log/nginx/api.access.log;
    error_log  /var/log/nginx/api.error.log;

    # 请求限流（防止刷接口）
    limit_req_zone $binary_remote_addr zone=api:10m rate=10r/s;
    limit_req zone=api burst=20 nodelay;

    # 反向代理 + 缓存
    location / {
        proxy_pass http://api_backend;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-Proto https;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;

        # 超时设置
        proxy_connect_timeout 5s;
        proxy_send_timeout 60s;
        proxy_read_timeout 60s;

        # 缓存
        proxy_cache my_cache;
        proxy_cache_valid 200 302 5m;
        add_header X-Cache-Status $upstream_cache_status;
    }

    # 健康检查端点（不缓存）
    location /health {
        proxy_pass http://api_backend;
        proxy_cache off;
        access_log off;
    }
}
```

## 七、常见问题与排查

### 1. 502 Bad Gateway
- 后端服务未启动或端口错误
- `proxy_pass` 地址拼写错误
- 防火墙阻止 Nginx → 后端通信

**排查命令**：
```bash
nginx -t          # 检查配置语法
systemctl status nginx
curl http://backend:8080/health  # 测试后端连通性
```

### 2. 负载不均衡
- 检查 `upstream` 算法是否设置正确
- 长连接可能导致分配不均，可调整 `keepalive` 参数

### 3. 缓存不生效
- 确认 `proxy_cache_path` 路径存在且 Nginx 有写入权限
- 检查 `proxy_cache_valid` 规则是否匹配响应码
- 使用 `curl -I` 查看 `X-Cache-Status` 头

### 4. SSL 握手慢
- 启用 `ssl_stapling`（OCSP 装订）
- 使用 ECDHE 密钥交换，减少计算开销
- 启用 HTTP/2（多路复用）

## 八、自动化脚本：一键扩容后端

当需要新增后端节点时，手动改配置太麻烦。这里是一个简单的**配置模板生成器**：

```python
#!/usr/bin/env python3
# generate_upstream.py

UPSTREAM_NAME = "api_backend"
BACKENDS = [
    "10.0.1.10:8080",
    "10.0.1.11:8080",
]

def render_upstream():
    lines = [f"upstream {UPSTREAM_NAME} {{", "    least_conn;"]
    for be in BACKENDS:
        lines.append(f"    server {be} max_fails=3 fail_timeout=30s;")
    lines.append("}\n")
    return "\n".join(lines)

if __name__ == "__main__":
    print(render_upstream())
```

将输出插入到 Nginx 配置文件，然后 `nginx -s reload` 即可生效。

## 结语

Nginx 的深度功能远不止这些。但反向代理 + 负载均衡 + SSL 是生产环境的核心三件套。

**记住**：
- 配置前先设计架构（多少后端？算法？健康检查？）
- 缓存和压缩能显著提升性能，但不是万能
- 日志和监控是运维的眼睛，别省掉
