前言：为什么我们需要 MCP？

2024 年底，Anthropic 推出了 Model Context Protocol (MCP)——一个开放标准，旨在让 LLM 能够安全、结构化地与外部工具和数据源交互。

在此之前，Function Calling 是主流方案，但它有几个痛点：

  • 协议私有：每家 LLM 厂商都有自己的格式
  • 生态割裂：工具描述无法复用
  • 安全性弱：缺少权限模型和资源隔离

MCP 的出现，就像 USB 对于外设的意义——统一接口，一次集成，处处可用。

MCP 架构浅析

MCP 采用 客户端-服务器 模型，通信基于 stdio 或 SSE，消息格式为 JSON-RPC 2.0：

核心概念：

  • Server：提供工具/资源，如文件系统、Git、数据库
  • Client：连接 Server 并转发 LLM 的请求（Claude Desktop 就是官方 Client）
  • Transport：stdio（本地）或 SSE（远程）
  • Prompts：预定义对话模板，用于引导 LLM

安全性考量

MCP 开放了工具调用能力，必须注意：

  1. 权限最小化：只暴露必要的工具和目录
  2. 输入验证：Sanitize 用户输入的路径和 SQL
  3. 沙箱运行：建议在容器内运行 MCP Server
  4. 审计日志：记录所有工具调用用于追踪

官方推荐使用 --read-only 模式启动文件系统 Server（如果实现支持）。

MCP vs Function Calling

特性 MCP Function Calling (OpenAI)
协议标准 开放标准 厂商私有
生态复用 ✅ 工具可跨 LLM 使用 ❌ 绑定特定模型
传输机制 stdio / SSE HTTP only
权限模型 资源级控制 无内置
社区支持 快速增长 成熟但封闭

结论：如果你在构建跨模型工具集，MCP 是更优选择；如果只在 OpenAI 生态内，Function Calling 仍可胜任。

小结

MCP 协议为 AI 工具集成提供了标准化、可复用、安全的解决方案。从简单的文件系统 Server 到复杂的企业级数据平台，MCP 都能胜任。
